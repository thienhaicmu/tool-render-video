# Task Definition Standard

## What Is a Task?

A task is a discrete unit of engineering work that can be:
- Described in a single objective sentence
- Scoped to a bounded set of files
- Reviewed by a single engineer in under 30 minutes
- Tested with concrete acceptance criteria

Tasks larger than this definition should be broken down.

## Raw Task Format

A raw task is a JSON file dropped into `tasks/incoming/`.
Required fields:
- `task_id`: unique identifier (generated by pipeline if absent)
- `submitted_by`: identity of submitter
- `title`: short descriptive title
- `raw_prompt`: the engineering request in natural language

Optional fields:
- `priority`: low | normal | high | critical
- `tags`: array of topic tags
- `context_hints`: paths or concepts to include in normalization context
- `target_area`: backend | frontend | infra | testing

## Normalized Task Format

After normalization, the task becomes a structured spec.
See `schemas/normalized-prompt.schema.json` for full field list.

Key fields:
- `task_type`: feature | bugfix | refactor | infra | test | docs | security | performance
- `objective`: single clear sentence
- `acceptance_criteria`: array of testable conditions
- `scope_in` / `scope_out`: explicit boundaries
- `estimated_complexity`: trivial | small | medium | large | epic

## Task Lifecycle

```
incoming → normalized → packaged → executing → executed → reviewed → archived
```

Each transition is logged as a structured event.

## Complexity Guidelines

| Level | LOC Changed | Review Time | Example |
|-------|-------------|-------------|---------|
| trivial | < 10 | < 5 min | Fix typo in error message |
| small | 10-50 | < 15 min | Add a new API field |
| medium | 50-200 | < 30 min | New endpoint with tests |
| large | 200-500 | < 60 min | New service module |
| epic | > 500 | > 60 min | Should be decomposed |
